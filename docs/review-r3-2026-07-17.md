## memory-safety — score 80/100

- [high] `scripts/01_memwatch.sh:123-127` treats any `TERM`, `INT`, or `HUP` as an expected exit and does not terminate the armed engine. An accidental HUP or supervisor signal therefore leaves inference unsupervised until the 60-second guard notices (`configs/systemd/dsv4-guard.timer:5-8`); memory can cross the UMA freeze boundary during that interval.

- [high] The watchdog starts first but remains unarmed until after the engine is launched and its child PID is discovered (`scripts/20_serve_ds4.sh:528-553`, `scripts/21_serve_llamacpp.sh:497-522`). While unarmed, a breach is merely logged (`scripts/01_memwatch.sh:224-227`). A fast startup allocation or delayed PID discovery can freeze the host before the target file is published.

- [medium] Failed-start cleanup terminates the watchdog before the engine (`scripts/20_serve_ds4.sh:73-85`, with the same ordering in `scripts/21_serve_llamacpp.sh:73-85`). A startup error under memory pressure creates a brief unsupervised interval during cleanup.

- [low] The important round-2 fixes otherwise hold: breach SIGKILL precedes logging (`scripts/01_memwatch.sh:207-223`), PID start ticks are checked before signaling (`scripts/01_memwatch.sh:63-98`), both stacks share `/run/dsv4/inference.lock`, and static budgets retain 16 GiB while the watchdog trips at 12 GiB. The ds4 soak’s 12.439 GiB minimum shows, however, that its observed margin above the kill line was only about 0.44 GiB (`results/soak-ds4.json:10-43`).
