# Executive summary
**Verdict: PROCEED-WITH-MITIGATIONS.**
The v0.2.2 fast-path artifacts are source-built in-process from the base GGUF; neither the build nor runtime downloads them.
No make-time fetch, Git submodule, production-server telemetry, analytics, or update check was found.
**HIGH:** all three default GGUFs come from mutable Hugging Face `main` URLs without SHA-256; the personal `bleysg` drafter repository is the highest-risk input.
The HTTP API has no authentication, but defaults to `127.0.0.1:8000` and honors `--host`; serving must remain explicitly loopback-only.
Pin engine commit `baa889025b16a7060f5f854226cb0d14e260eb52`, not tag `v0.2.2`, and pin recipe commit `039e047309f5acbf05c1aa3f4221f95cc00e09bb`.
Require a SHA-256 allowlist, a private lock/runtime directory, no CORS, and no distributed/agent/web mode.
Do not deploy until immutable HF revisions and the three expected GGUF digests are independently established.

## B. v0.2.2 “always-on fast-path artifacts”

### B4. What the artifacts are and how they are produced

The artifacts are **derived GPU-resident aligned weight layouts**, not prebuilt binaries, downloaded kernels, cache files, or remote objects. They comprise aligned-SoA repacks for IQ2_XXS routed-expert gate/up tensors, row-pair SoA Q2_K routed-expert down tensors, and aligned Q8_0 dense tensors (`vendor/ds4/cuda/mmq/ds4_repack.cu:442`, `vendor/ds4/cuda/mmq/ds4_repack.cu:456`, `vendor/ds4/cuda/mmq/ds4_repack.cu:865`). v0.2.2 moved the builders from `ds4_weight_server` into shared, compiled source `cuda/mmq/ds4_repack.{h,cu}`, links `ds4_repack.o` into both engine and weight server, and changed standalone engine open to call the builder before mapping the base model (`vendor/ds4/CHANGELOG.md:14`, `vendor/ds4/CHANGELOG.md:21`, `vendor/ds4/Makefile:39`, `vendor/ds4/Makefile:255`, `vendor/ds4/Makefile:258`, `vendor/ds4/ds4.c:36457`, `vendor/ds4/ds4.c:36466`).

Mechanism, quoted compactly from code: precedence is “`manifest import > in-process build > raw`”; the builder reopens `model_path` read-only, parses the local GGUF catalog, then builds in order `q2k, iq2, q8` with CUDA kernels and registers the resulting device pointers (`vendor/ds4/ds4_cuda.cu:4070`, `vendor/ds4/ds4_cuda.cu:4078`, `vendor/ds4/ds4_cuda.cu:4113`, `vendor/ds4/ds4_cuda.cu:4123`, `vendor/ds4/ds4_cuda.cu:4157`, `vendor/ds4/ds4_cuda.cu:4182`). The shared source opens the GGUF `O_RDONLY`, optionally reopens it through `/proc/self/fd/...` with `O_DIRECT`, and maps it read-only; it has no downloader or artifact-file writer (`vendor/ds4/cuda/mmq/ds4_repack.cu:52`, `vendor/ds4/cuda/mmq/ds4_repack.cu:53`, `vendor/ds4/cuda/mmq/ds4_repack.cu:68`, `vendor/ds4/cuda/mmq/ds4_repack.cu:71`, `vendor/ds4/cuda/mmq/ds4_repack.cu:82`). The release documents 78.7 GiB built in about 22–26 seconds on GB10 (`vendor/ds4/CHANGELOG.md:23`).

The FNV-1a mechanism is a developer bit-identity gate, not provenance verification: `DS4_WS_REPACK_HASH=1` optionally hashes produced artifact bytes, and the release reports 474/474 equality against weight-server output (`vendor/ds4/cuda/mmq/ds4_repack.cu:481`, `vendor/ds4/cuda/mmq/ds4_repack.cu:483`, `vendor/ds4/cuda/mmq/ds4_repack.cu:496`, `vendor/ds4/cuda/mmq/ds4_repack.cu:524`, `vendor/ds4/CHANGELOG.md:27`). Trust still rests on the pinned source and cryptographic verification of the input GGUF.

The diff also adds observability: a canonical boot line reports `built`, `imported`, or raw-tier `none`, while `/metrics` and `/v1/stats` expose source/count/bytes (`vendor/ds4/ds4_cuda.cu:4235`, `vendor/ds4/ds4_cuda.cu:4241`, `vendor/ds4/ds4_cuda.cu:4248`, `vendor/ds4/ds4_server.c:12978`, `vendor/ds4/ds4_server.c:13006`). The only unrelated engine behavior in the diff is the `enable_thinking` request alias (`vendor/ds4/CHANGELOG.md:39`).

### B5. Host/provenance risk and source-only path

The always-on aligned artifacts are **not fetched from any host**, so the requested third-party-artifact HIGH-RISK condition does not apply to them. Source-only in-process production is the default. `DS4_CUDA_BUILD_ARTIFACTS=0` disables only boot-time construction and falls back to the slower raw layout; `DS4_CUDA_NO_DERIVED_WEIGHTS` disables derived artifacts; a configured `DS4_CUDA_WEIGHT_IPC_MANIFEST` instead imports local weight-server allocations (`vendor/ds4/ds4_cuda.cu:4085`, `vendor/ds4/ds4_cuda.cu:4090`, `vendor/ds4/ds4_cuda.cu:4091`, `vendor/ds4/ds4_cuda.cu:4095`). Reimplementation decision: **retain the default source-built path, explicitly set `DS4_CUDA_BUILD_ARTIFACTS=1`, do not configure a weight-server manifest, and verify the boot line/stats say `built`.**

Do not confuse these repacks with the **separate DSpark drafter GGUF**. That 6.5 GiB prebuilt model is downloaded from the personal `bleysg` HF repository without a digest and is HIGH RISK (`vendor/ds4-on-spark/README.md:243`, `vendor/ds4-on-spark/README.md:245`, `vendor/ds4-on-spark/README.md:246`). A local extraction path exists from official checkpoint shards, but the included script builds Q4_K or Q8_0 experts, not the installer's exact Q2K artifact: it reads a local safetensors index/shards and exposes `--experts q4_k|q8_0` (`vendor/ds4/gguf-tools/dspark_extract.py:2`, `vendor/ds4/gguf-tools/dspark_extract.py:19`, `vendor/ds4/gguf-tools/dspark_extract.py:63`, `vendor/ds4/gguf-tools/dspark_extract.py:67`, `vendor/ds4/gguf-tools/dspark_extract.py:204`, `vendor/ds4/gguf-tools/dspark_extract.py:207`). The recipe repo documents the Q4_K command (`vendor/ds4-on-spark/README.md:248`, `vendor/ds4-on-spark/README.md:252`). Thus there is no locally documented source-reproduction path for the exact Q2K drafter.

## C. Engine code at v0.2.2

### C6. Network surface

**Production `ds4-server` path.** Normal single-host server operation creates only its inbound HTTP listener. No outbound HTTP, telemetry, analytics, or update check was found. The listener is described in C7. Two opt-in transport paths are compiled in:

- **Distributed TCP:** `--role worker --coordinator HOST PORT` resolves and connects to the configured coordinator and reconnects indefinitely; workers also open a TCP data listener, while coordinators listen for registrations (`vendor/ds4/ds4_distributed.c:1221`, `vendor/ds4/ds4_distributed.c:1232`, `vendor/ds4/ds4_distributed.c:1242`, `vendor/ds4/ds4_distributed.c:1248`, `vendor/ds4/ds4_distributed.c:7651`, `vendor/ds4/ds4_distributed.c:7657`, `vendor/ds4/ds4_distributed.c:7701`, `vendor/ds4/ds4_distributed.c:7702`). The protocol has no encryption/authentication and is out of scope for the recommended single-host launch.
- **Weight-server IPC:** when a manifest is configured, CUDA reads it and connects to a path named in it using `AF_UNIX`, then requests exported CUDA allocation FDs; this is local Unix IPC, not IP egress (`vendor/ds4/ds4_cuda.cu:3460`, `vendor/ds4/ds4_cuda.cu:3462`, `vendor/ds4/ds4_cuda.cu:3467`, `vendor/ds4/ds4_cuda.cu:3476`, `vendor/ds4/ds4.c:36568`). - **Bind:** default `127.0.0.1`, port `8000`; `--host HOST` and `--port N` are honored (`vendor/ds4/ds4_server.c:13536`, `vendor/ds4/ds4_server.c:13544`, `vendor/ds4/ds4_server.c:13545`, `vendor/ds4/ds4_server.c:13592`, `vendor/ds4/ds4_server.c:13594`). `localhost` is normalized to `127.0.0.1`, and only numeric IPv4 accepted by `inet_pton` is supported (`vendor/ds4/ds4_server.c:13289`, `vendor/ds4/ds4_server.c:13299`, `vendor/ds4/ds4_server.c:13300`, `vendor/ds4/ds4_server.c:13305`). `--cors` adds permissive browser headers but does not change the bind (`vendor/ds4/ds4_server.c:13458`, `vendor/ds4/ds4_server.c:13462`).
- **Authentication:** none. The parsed request structure retains method, path, body, and an `Accept: application/json` boolean only; header parsing recognizes `Accept` and `Content-Length`, not `Authorization`, bearer tokens, or API keys (`vendor/ds4/ds4_server.c:12570`, `vendor/ds4/ds4_server.c:12575`, `vendor/ds4/ds4_server.c:12593`, `vendor/ds4/ds4_server.c:12609`, `vendor/ds4/ds4_server.c:12626`, `vendor/ds4/ds4_server.c:12649`, `vendor/ds4/ds4_server.c:12655`). - **Endpoints:** `OPTIONS *`; `GET /metrics`; `GET /v1/stats`; `GET /v1/models`; `GET /v1/models/deepseek-v4-flash`; `GET /v1/models/deepseek-v4-pro`; `POST /v1/batch`; `POST /v1/messages`; `POST /v1/chat/completions`; `POST /v1/responses`; and `POST /v1/completions` (`vendor/ds4/ds4_server.c:13148`, `vendor/ds4/ds4_server.c:13154`, `vendor/ds4/ds4_server.c:13159`, `vendor/ds4/ds4_server.c:13165`, `vendor/ds4/ds4_server.c:13170`, `vendor/ds4/ds4_server.c:13181`, `vendor/ds4/ds4_server.c:13191`, `vendor/ds4/ds4_server.c:13194`, `vendor/ds4/ds4_server.c:13197`, `vendor/ds4/ds4_server.c:13200`). The README documents all except `/v1/batch` (`vendor/ds4/README.md:734`, `vendor/ds4/README.md:743`). Unknown routes return 404 (`vendor/ds4/ds4_server.c:13203`).

### C8. Runtime file writes

- **Always:** every `ds4_engine_open` acquires `DS4_LOCK_FILE`, default `/tmp/ds4.lock`, with `open(O_RDWR|O_CREAT,0600)`, then truncates and writes the PID (`vendor/ds4/ds4.c:27418`, `vendor/ds4/ds4.c:27421`, `vendor/ds4/ds4.c:27422`, `vendor/ds4/ds4.c:27424`, `vendor/ds4/ds4.c:27454`, `vendor/ds4/ds4.c:27459`, `vendor/ds4/ds4.c:36370`). It does not use `O_NOFOLLOW` or unlink the file at exit. A pre-created symlink in shared `/tmp` could therefore redirect truncation to another file writable by the serving user. Set `DS4_LOCK_FILE` inside a mode-0700 runtime directory.
- **Model and fast artifacts:** base, MTP, and DSpark models are opened/mapped read-only; fast repacks remain device allocations (`vendor/ds4/ds4.c:1589`, `vendor/ds4/ds4.c:1594`, `vendor/ds4/cuda/mmq/ds4_repack.cu:53`, `vendor/ds4/ds4_cuda.cu:4182`).
- **Server defaults:** no KV directory and no trace file are configured in the default struct; they are created only when `--kv-disk-dir DIR` or `--trace FILE` is passed (`vendor/ds4/ds4_server.c:13339`, `vendor/ds4/ds4_server.c:13340`, `vendor/ds4/ds4_server.c:13550`, `vendor/ds4/ds4_server.c:13598`, `vendor/ds4/ds4_server.c:13600`, `vendor/ds4/ds4_server.c:13719`, `vendor/ds4/ds4_server.c:13733`). Trace opens the caller-selected path with mode `w` and records prompts/cache decisions/output/tool calls per the help (`vendor/ds4/ds4_server.c:13464`, `vendor/ds4/ds4_server.c:13465`, `vendor/ds4/ds4_server.c:13734`).
- **Disk KV if enabled:** recursively creates the supplied directory with mode 0700, scans/updates `<SHA1>.kv`, evicts files to respect the budget, writes same-directory `.tmp.<pid>` then renames atomically, and can persist on cold/continued/evict/shutdown events (`vendor/ds4/ds4_kvstore.c:340`, `vendor/ds4/ds4_kvstore.c:355`, `vendor/ds4/ds4_kvstore.c:361`, `vendor/ds4/ds4_kvstore.c:367`, `vendor/ds4/ds4_kvstore.c:589`, `vendor/ds4/ds4_kvstore.c:609`, `vendor/ds4/ds4_kvstore.c:619`, `vendor/ds4/ds4_kvstore.c:1054`, `vendor/ds4/ds4_kvstore.c:1058`, `vendor/ds4/ds4_kvstore.c:1107`). KV files include rendered prompt text and serialized model state (`vendor/ds4/README.md:1044`, `vendor/ds4/README.md:1060`, `vendor/ds4/README.md:1088`). Serialization also stages a temporary payload at hard-coded `/tmp/ds4-session-payload.XXXXXX` and removes it afterward (`vendor/ds4/ds4.c:30555`, `vendor/ds4/ds4.c:30558`, `vendor/ds4/ds4.c:30576`, `vendor/ds4/ds4.c:30577`, `vendor/ds4/ds4.c:30610`). Thus enabled KV can write outside both cwd and the selected KV directory.
- **Caller-selected diagnostic writes:** CLI options and debug environment variables can write requested logits/imatrix/dump paths; for example `DS4_DUMP_FIRST_LOGITS` opens its supplied path with `wb` (`vendor/ds4/ds4.c:19303`, `vendor/ds4/ds4.c:19306`, `vendor/ds4/ds4.c:19309`), and `--imatrix-out` writes its supplied path (`vendor/ds4/ds4.c:20298`, `vendor/ds4/ds4.c:20302`). None is enabled by the recommended server command.

There is no `ds4_ssd.c` in the v0.2.2 tree. The relevant persistent cache implementation is `ds4_kvstore.c`; no implicit SSD cache directory was found.

### C9. Credential, dotfile, SSH, and system reads

The production `ds4-server` code does not read API keys, SSH files, cloud credentials, `$HOME` dotfiles, or `/etc` files. Its environment reads are DS4 configuration/performance knobs and model/manifest paths. It does read `/proc/meminfo` for memory sizing and `/proc/self/fd/<n>` for optional direct I/O (`vendor/ds4/ds4.c:1536`, `vendor/ds4/ds4_cuda.cu:3405`, `vendor/ds4/ds4_cuda.cu:3407`).

- Manual official-quality collectors read `DEEPSEEK_API_KEY` and send it as a bearer token (C6 citations).
- `ds4_web` reads `HOME` and creates `$HOME/.ds4/browser`, but is linked to `ds4-agent`, not `ds4-server` (`vendor/ds4/ds4_web.c:1213`, `vendor/ds4/ds4_web.c:1217`, `vendor/ds4/Makefile:121`, `vendor/ds4/Makefile:130`). No `~/.ssh`, SSH key, `/etc/passwd`, `.aws`, `.netrc`, or similar credential-path read was found in the audited C/CUDA/shell production paths.

`make` performs local compilation/linking only. ## E. Verdict and reimplementation specification

### E12. Findings by severity

1. **HIGH — unverified mutable model downloads.** All three URLs use `/resolve/main/`; existing files are checked only by `Content-Length`, while new/resumed files receive no post-download check and no hash (`vendor/ds4-on-spark/install.sh:247`, `vendor/ds4-on-spark/install.sh:249`, `vendor/ds4-on-spark/install.sh:251`, `vendor/ds4-on-spark/install.sh:254`, `vendor/ds4-on-spark/install.sh:262`). A compromised account/CDN, moved `main`, corrupt resume prefix, or same-length substitution is accepted.
2. **HIGH — personal-repository DSpark drafter.** The default 6.5 GiB Q2K drafter comes from `bleysg/...`, not the pinned Git repository or `antirez` model repository, and has no local checksum/provenance record (`vendor/ds4-on-spark/install.sh:63`, `vendor/ds4-on-spark/install.sh:64`, `vendor/ds4-on-spark/install.sh:276`, `vendor/ds4-on-spark/README.md:243`, `vendor/ds4-on-spark/README.md:245`). The exact Q2K file is not reproducible with the included Q4_K/Q8_0-only extractor (`vendor/ds4/gguf-tools/dspark_extract.py:207`). Do not consume it until an immutable revision and independently trusted SHA-256 are approved.
3. **MEDIUM — unauthenticated HTTP API if exposed beyond loopback.** Request parsing implements no Authorization/API-key check and routes inference, batch, metrics, and stats endpoints directly (`vendor/ds4/ds4_server.c:12570`, `vendor/ds4/ds4_server.c:12626`, `vendor/ds4/ds4_server.c:13154`, `vendor/ds4/ds4_server.c:13181`, `vendor/ds4/ds4_server.c:13191`). Default loopback materially mitigates this (`vendor/ds4/ds4_server.c:13544`); `--host 0.0.0.0` would expose unauthenticated, expensive inference. Use a loopback reverse proxy with authentication if remote clients are required.
4. **MEDIUM — installer Git pin is a mutable name, not verified commit.** Clone/fetch uses `DS4_REF=v0.2.2`, then hard-resets to whatever was fetched, with no comparison to `baa889...` and no signature verification (`vendor/ds4-on-spark/install.sh:49`, `vendor/ds4-on-spark/install.sh:210`, `vendor/ds4-on-spark/install.sh:220`, `vendor/ds4-on-spark/install.sh:221`). Pin and verify the full commit SHA.
5. **MEDIUM — predictable shared lock path and symlink-following truncate.** Every engine run opens `/tmp/ds4.lock` without `O_NOFOLLOW` and truncates it (`vendor/ds4/ds4.c:27421`, `vendor/ds4/ds4.c:27422`, `vendor/ds4/ds4.c:27424`, `vendor/ds4/ds4.c:27454`). Set `DS4_LOCK_FILE` to a mode-0700 directory owned by the service user.
6. **MEDIUM — `--no-dspark` does not mean plain decode.** It leaves default MTP enabled, downloads MTP, and starts MTP-2 if present (`vendor/ds4-on-spark/install.sh:78`, `vendor/ds4-on-spark/install.sh:118`, `vendor/ds4-on-spark/install.sh:272`, `vendor/ds4-on-spark/install.sh:318`, `vendor/ds4-on-spark/install.sh:320`). Reimplementation must select plain/MTP/DSpark explicitly rather than copy this state machine.
7. **LOW — fast-path artifacts are local source-derived and observable.** They are built from the verified base GGUF by pinned CUDA source, can be disabled, and report their source; no external artifact fetch exists (`vendor/ds4/ds4_cuda.cu:4070`, `vendor/ds4/ds4_cuda.cu:4080`, `vendor/ds4/ds4_cuda.cu:4095`, `vendor/ds4/ds4_cuda.cu:4235`). The optional FNV check is reproducibility evidence, not cryptographic provenance (`vendor/ds4/cuda/mmq/ds4_repack.cu:483`, `vendor/ds4/cuda/mmq/ds4_repack.cu:524`).
8. **LOW — optional paths can persist sensitive prompt/session data.** `--trace` records prompts/output/tool calls, and disk KV files contain rendered prompts; KV staging uses `/tmp` (`vendor/ds4/ds4_server.c:13464`, `vendor/ds4/ds4_server.c:13465`, `vendor/ds4/README.md:1088`, `vendor/ds4/ds4.c:30576`). Leave both disabled unless required; if enabled, use a private directory and retention policy.
9. **LOW — opt-in network tooling is broad but not automatic.** Distributed mode opens/connects TCP, `ds4-agent` directs Chrome to the web, test/benchmark scripts use SSH/curl, and manual quality tools call DeepSeek with an API key (C6). None runs in the recommended server command, and no analytics/update mechanism was found.
10. **LOW — installer side-effect/build documentation is incomplete.** `make cuda` builds three additional binaries, `/tmp/ds4.lock` is written, and `$HOME/ds4-server.log` is outside the two headline directories (`vendor/ds4/Makefile:116`, `vendor/ds4/ds4.c:27422`, `vendor/ds4-on-spark/install.sh:328`). No sudo/service/cron/rc/PATH modification occurs.

### E14. Required patch/configuration for safe serving

No engine source patch is required for **strict loopback-only** serving. Required configuration is: explicit `--host 127.0.0.1`, no `--cors`, a private `DS4_LOCK_FILE`, verified read-only weights, and no distributed/agent/weight-server options. There is no unauthenticated `/shutdown` endpoint to disable. `/metrics`, `/v1/stats`, `/v1/batch`, and generation endpoints are unauthenticated, so **never bind ds4 directly to a LAN/WAN interface**. If direct non-loopback binding is a hard requirement, patching in constant-time bearer-token validation before every route or placing a proven authenticated reverse proxy in front becomes mandatory; the current engine has no API-key flag.

Leave trace and disk KV disabled in the baseline. If KV is required, use a dedicated mode-0700 directory, account for the hard-coded `/tmp` staging file, and document that cached files contain prompt text. A defense-in-depth service sandbox should deny outbound IP network access; this does not interfere with the recommended single-host server path and prevents accidental activation of distributed or agent-like egress.
