# Production security model

The selected inference stack is reachable remotely only through Tailscale
Serve. Every application listener remains bound to loopback on the Spark.

```text
tailnet client
  |  tailnet identity + Tailscale HTTPS
  v
tailscaled on Spark
  |-- ds4 winner ------> 127.0.0.1:8010 Caddy --forward_auth--> :8014 helper
  |                                             |
  |                                             +--success----> :8012 ds4
  |
  +-- llama.cpp winner -> 127.0.0.1:8011 llama-server (native API key)
```

## Loopback exposure

| Port | Process | Authentication | Exposure |
|---:|---|---|---|
| 8010 | Caddy for ds4 | Bearer token on every path | Tailscale Serve target for ds4 |
| 8012 | ds4 engine | None | Internal loopback backend only |
| 8014 | auth helper | Exactly one bearer header | Internal Caddy subrequest only |
| 8011 | llama.cpp | Native API key, except public metadata | Tailscale Serve target for llama.cpp |

Tailscale supplies encrypted, tailnet-only transport and device/user policy.
It does not replace the application bearer key. Tailscale Serve must target
only port 8010 for ds4 or port 8011 for llama.cpp; never target 8012 or 8014.
No listener is intended for a LAN or WAN address.

llama.cpp intentionally leaves `/health` (and related health endpoints) and
`/v1/models` unauthenticated. Treat their metadata as visible to authorized
tailnet members. The ds4 engine also has unauthenticated metadata and inference
routes, but Caddy authenticates **all** paths before forwarding anything to it
and strips the `Authorization` header before the upstream request.

## Key handling and rotation

The installer creates a 256-bit production key at
`/etc/deepseek-v4-flash/api-key` only when it is absent. The file is
`root:dsv4` mode `0640`; its directory is mode `0750`. The adjacent `env` file
contains only the key path, not the key. The helper reads the key once at start
and again on SIGHUP. llama.cpp reads it when the server starts. Neither service
logs the Authorization value.

To rotate, repeat the installer's key-generation step using a root-only
temporary file, set it to `root:dsv4` mode `0640`, atomically replace
`api-key`, and restart the installed winner (plus `dsv4-authhelper.service` for
ds4). Then replace the saved credential on every authorized laptop. llama.cpp
clients must likewise replace any earlier gate-phase key with the production
key generated by the installer. Never print or pass the token on a process
command line outside a short-lived local verification shell.

## Process and network isolation

The `dsv4` account owns the engine builds, model working data, logs, and
runtime state; it is not a login or administrative identity. `ProtectHome=no`
is deliberate because `/home/dsv4` is that working area, while
`ProtectSystem=strict` and `ReadWritePaths` keep persistent writes confined to
it. `/run/dsv4` holds only volatile locks and state.

All units deny IP traffic by default and allow only IPv4/IPv6 loopback, with
address families restricted to IP and Unix sockets. This is defense in depth
against accidental distributed, agent, or weight-server egress described in
the ds4 security review. It also prevents either engine or proxy from becoming
an outbound network client while preserving the single-host serving path.

The inference units do not automatically restart. The serve scripts enforce
the authoritative memory budget and residency lock, and their OOM watchdog can
kill an unsafe run. After a watchdog event, an operator reviews the logs and
restarts the winner manually according to the runbook.
